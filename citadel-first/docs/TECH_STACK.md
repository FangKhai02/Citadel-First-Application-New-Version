# Citadel First — Technology Stack & Project Reference

> **For Claude Code sessions:** Load this file at the start of every session to understand the project architecture, tech stack decisions, and database schema. When implementing any feature, cross-reference the **Database Schema Reference** section to identify the correct PostgreSQL tables and fields. You MUST follow the **Coding Standards** section — the owner prioritises clean, well-structured code above all else.

---

## Project Overview

**Citadel First** is a wealth management and trust product investment platform with three components:

| Component | Technology | Purpose |
|-----------|-----------|---------|
| Mobile App | Flutter (Dart) | Clients buying trust products + Agents selling them (single app, role-based) |
| Backend API | FastAPI (Python 3.11+) | REST API with auto-generated Swagger UI |
| Admin Web | Next.js 14 (TypeScript) | Internal staff: approvers, finance, CTB/CWP admins |
| Database | PostgreSQL | Existing schema — do NOT recreate tables |

---

## Tech Stack Decisions

| Layer | Technology | Why |
|-------|-----------|-----|
| Backend | **FastAPI** | Auto-generates Swagger/OpenAPI at `/docs`, async-first, Pydantic v2 type safety, excellent PostgreSQL support |
| ORM | **SQLAlchemy + Alembic** | Industry standard for Python + PostgreSQL, full migration support |
| Auth | **JWT (python-jose)** | Stateless, works with both Flutter and Next.js |
| Password | **bcrypt (passlib)** | Matches existing password hashes in DB (`$2y$10$...` format) |
| Mobile | **Flutter + Bloc** | Cross-platform iOS/Android, client-requested, Bloc for predictable state |
| Admin Web | **Next.js 14 + shadcn/ui** | App Router, TypeScript, TailwindCSS, TanStack Query |
| File Storage | **AWS S3** | Documents, identity images, bank proofs — DB stores only S3 object key (`_key` suffix fields) |
| API Docs | **Swagger UI** | Auto-generated by FastAPI at `http://localhost:8000/docs` |
| Dev | **Docker Compose** | Local PostgreSQL + backend + admin web |

> **NOT used:** Laravel (previous vendor, replaced), React Native (mobile framework, not backend — FastAPI is the backend)

## Coding Standards

> The owner's priority is **clean, well-structured code**. Every AI session must follow these rules without being asked.

### General Rules

- **Every feature that needs data must call the API** — no hardcoded data, no mock data in production code, no direct DB queries from the Flutter app
- **No business logic in UI widgets** — widgets only display state; all logic lives in Blocs (Flutter) or services (FastAPI)
- **One responsibility per file** — a screen file contains only the UI, a Bloc file contains only state logic, a service file contains only business logic
- **No unused code** — do not leave commented-out code, unused imports, or placeholder variables in committed files
- **No magic numbers or strings** — colours go in constants, URLs go in `ApiEndpoints`, DB table names go in SQLAlchemy models

### Flutter (Mobile) Standards

| Rule | Detail |
|------|--------|
| **All API calls go through `ApiClient`** | Never use `http` or raw `Dio` directly in a widget or Bloc — always use the shared `ApiClient` singleton |
| **Bloc for every feature** | Each screen/feature has its own `*_bloc.dart`, `*_event.dart`, `*_state.dart` — no `setState` for business data |
| **`ApiEndpoints` for all URLs** | Never write a URL string in a Bloc or widget — add it to `lib/core/api/api_endpoints.dart` first |
| **Secure storage for tokens** | Never store JWT tokens in `SharedPreferences` — always use `SecureStorage` wrapper |
| **Named routes only** | All navigation uses `context.go('/route')` via go_router — no `Navigator.push` |
| **Error handling in Bloc** | Catch `DioException` in the Bloc, emit a failure state with a message — never show raw exception text to the user |

### FastAPI (Backend) Standards

| Rule | Detail |
|------|--------|
| **Route files are thin** | Routes only validate input and call a service — no DB queries inside route handlers |
| **Services hold logic** | All business logic (calculations, multi-step DB operations) goes in `app/services/` |
| **Pydantic schemas for everything** | Every request body and every response must have a Pydantic schema — no raw `dict` returns |
| **Async everywhere** | All DB operations use `await` with `AsyncSession` — never use synchronous SQLAlchemy |
| **DECIMAL for money** | All financial fields use `DECIMAL` in SQLAlchemy (`Numeric` type) — never `Float` or `Double` |
| **Soft delete only** | Never run `DELETE` on business tables — set `is_deleted = 1` |
| **Parameterised queries only** | Never build SQL strings manually — always use SQLAlchemy ORM or `text()` with bound params |

### API Contract Rules

- Every new API endpoint must appear in Swagger UI (`/docs`) with a clear `summary` and correct response schema
- Mobile calls backend — mobile **never** queries PostgreSQL directly
- Admin web calls the same backend API as mobile — no separate admin database queries
- All endpoints are versioned under `/api/v1/` — never add routes outside this prefix

## Database Schema Reference

> The PostgreSQL database is already set up. SQLAlchemy models must reflect the existing schema — do NOT add new columns without a migration. All financial fields use `DECIMAL` (never FLOAT/DOUBLE). Files are stored in S3; the DB stores only the S3 object key (fields ending in `_key`).

| Mobile login (clients + agents) | `app_users` | `email_address`, `password` (bcrypt `$2y$`), `user_type` (CLIENT/AGENT/CORPORATE), `is_deleted` |

## Known Issues / Technical Debt

### Session-Based User Tracking for Signup Flow

**Issue:** All anonymous users share the same `app_user_id = 0` during the signup flow.

**Affected endpoints** (all use `_ANON_USER_ID = 0`):
- `POST /signup/bankruptcy-declaration` → `bankruptcy_declarations.user_id`
- `POST /signup/disclaimer-acceptance` → `disclaimer_acceptances.user_id`
- `POST /signup/trust-form-b6` → `trust_form_b6.user_id`
- `PATCH /signup/identity-document` → `user_details.app_user_id`
- `POST /signup/identity-document` → `user_details.identity_doc_type`
- `POST /signup/ocr` → returns OCR results (no DB storage)

**Impact:** When multiple users/testers go through the signup flow simultaneously, their data overwrites each other since all requests use `app_user_id = 0`.

**Root cause:** No session/session tracking between the mobile app and backend during the anonymous signup phase.

**Fix (TODO):**
1. **Backend:** Create a `signup_sessions` table with UUID-based session tracking:
   - `id` (UUID primary key)
   - `created_at`, `expires_at`
2. **New endpoint:** `POST /signup/sessions` to create a session
3. **Mobile:** Generate UUID when signup starts, store in SharedPreferences, send `X-Session-ID` header with all signup API calls
4. **Backend:** Modify signup endpoints to accept `X-Session-ID` header, look up session to get real `app_user_id`

**Status:** Deferred — currently only affects multi-user testing scenarios. Single-user testing works correctly.

## Key Business Rules

1. **Mobile user types:** `app_users.user_type` = `CLIENT` | `AGENT` | `CORPORATE` — determines Flutter app routing after login
2. **Order approval flow:** Checker → Finance → Approver (3 stages, each with status/remark/approver fields on `product_order`)
3. **Agent hierarchy:** `agent.recruit_manager_id` is self-referential (6 levels: Manager, P2P, SM, AVP, VP, SVP) — cascade commissions up the hierarchy
4. **Agency-product gating:** Always check `agency_product` before displaying products to an agent/client — not all products available to all agencies
5. **Soft deletes only:** `is_deleted` flag on most tables — never run hard DELETE
6. **Financial data type:** All money and percentages must use `DECIMAL` — the old schema used DOUBLE (wrong), do not repeat this
7. **S3 for files:** Database stores `_key` fields (S3 object keys), never binary data
8. **PIN ≠ Password:** Agents and clients each have a 4-digit PIN separate from login password
9. **Corporate dual signature:** `client_two_signature` table enforces two-signatory approval for corporate clients
10. **Products:** Current products include Vanguard, NIU, CTB — each may have different commission tiers

## Authentication Flow

### Mobile (Flutter)

```
POST /api/v1/auth/login
  body: { email: app_users.email_address, password: plaintext }
  → verify bcrypt against app_users.password
  → return { access_token, refresh_token, user_type }
  → Flutter routes based on user_type:
      CLIENT   → /client/dashboard
      AGENT    → /agent/dashboard
      CORPORATE → /client/dashboard (corporate view)
```

### Web Admin (Next.js)

```
POST /api/v1/auth/admin/login
  body: { email: users.email, password: plaintext }
  → verify bcrypt against users.password
  → load users.role_id → roles → permissions
  → return { access_token, role, permissions }
  → Next.js routes based on role:
      admin / CTB Admin → full access
      CTB Finance       → orders + commissions + dividends
      CTB Approver      → order approval queue
      CWP Approver      → CWP order approval
      CTB General Admin → read + client management
```
